# 📖 야구사전 콘텐츠 슬래시 명령 스킬 — `/야구사전`

사용자가 `/야구사전 {용어}`를 입력하면 해당 야구 용어/규칙에 대해
**리서치 노트 + 말리 롱폼 대본 + SEO 패키지** 세 가지를 한 번에 만든다.
모든 출력은 **단일 파일** `50_Outputs/야구사전/{YYYY-MM-DD}_{용어}.md` 하나에
세 섹션(🔍 리서치 / 🎙 대본 / 🏷 SEO)으로 저장한다.

이 스킬은 사용자가 이미 Obsidian vault(보통 `C:\Users\user\Documents\Joy_SecondBrain` 또는
유사 위치)를 마운트해 둔 상태를 전제로 한다. 마운트 위치가 다르면 마운트된 폴더를 자동 사용한다.

> 🧩 1단계 범위: 리서치 + 대본 + SEO 텍스트까지만 생성한다.
> 이미지·영상 등 에셋 생성은 이후 단계에서 STEP으로 추가 예정 (아래 "확장 예정" 참고).

## 🚀 트리거 감지 규칙

| 입력 패턴 | 동작 |
|-----------|------|
| `/야구사전 보크` | "보크" 용어로 실행 |
| `/야구사전 인필드 플라이` | 여러 단어 용어도 전체를 하나의 용어로 인식 |
| `/야구사전 스트라이크존 규칙` | 규칙 주제도 동일하게 처리 |
| `/야구사전` (용어 없음) | 사용자에게 용어를 1회 질문 |
| `/야구사전 보크 짧게` | "짧게" 힌트 → 대본 3분 하한 기준으로 압축 |
| `/야구사전 보크 대본만` | 해당 결과물만 생성 (아래 주의사항 참고) |

## 입력 파싱 우선순위

1. **용어/규칙**: `/야구사전` 뒤의 텍스트 전체를 용어로 간주 (예약어 제외)
2. **예약어**: "짧게"(3분 기준), "길게"(5분 기준), "대본만", "리서치만", "SEO만"
3. 용어가 없으면 사용자에게 1회 질문. 용어가 모호하면(예: "플라이" 단독)
   후보 2~3개를 제시하고 확인받는다.

## 전체 워크플로 (순서대로 실행)

### STEP 0 — 실행 확인 메시지

사용자에게 짧게 실행 시작을 알린다.

```
📖 야구사전 "{용어}" 콘텐츠를 생성합니다...
   1) 리서치 (기원·역사·KBO 사례)  2) 말리 롱폼 대본  3) SEO 패키지
```

### STEP 1 — 리서치 (기원·역사·KBO 사례)

`web_search`로 해당 용어를 다각도로 조사한다.

검색 쿼리 예시:
- `{용어} 야구 뜻 규칙`
- `{용어} 유래 기원 어원`
- `{용어} KBO 사례 논란`
- `{용어} baseball rule origin history` (영문 교차 검증)

**수집할 정보 (최소):**
- 정확한 정의 (KBO 야구규칙 조항 번호가 있으면 명시)
- 기원·역사 (언제, 어디서, 왜 생겼는지 — MLB 유래면 도입 연도·배경)
- 용어의 어원 (영어 원어의 뜻, 한국식 표현이면 그 유래)
- KBO 실제 사례 2~3개 (유명 사건, 판정 논란, 경기 일자·팀·선수 포함)
- 흔한 오해나 헷갈리는 포인트 1~2개 (초보 팬이 자주 틀리는 부분)
- 관련 규칙·용어 (대본 마무리에서 다음 편 예고로 활용)

데이터가 부족하거나 출처끼리 충돌하면 `web_search`를 한 번 더 좁혀 재확인한다.
연도·사례·규칙 조항을 만들어내지 않는다. 확인 불가면 "확인 필요"로 표기한다.

**리서치 노트 구조 (단일 파일의 `## 🔍 리서치` 섹션):**

```
## 🔍 리서치

### 정의
(KBO 규칙 조항 번호 포함, 한 문단)

### 기원·역사
(연도·배경·도입 경위, 출처 함께 메모)

### 어원
(원어 뜻, 한국식 표현 유래)

### KBO 사례
| 일자 | 경기 | 내용 |
|---|---|---|
| YYYY-MM-DD | 팀A vs 팀B | (사건 요약) |

### 흔한 오해
- (오해 1) → (사실)
- (오해 2) → (사실)

### 관련 용어 (다음 편 후보)
- (용어 1), (용어 2)

### 출처
- (URL 목록)
```

### STEP 2 — Joy vault 스타일 가이드 자동 참조

가능하면 사용자의 vault에서 다음 파일을 `Read`로 펼쳐 본다(없거나 접근 불가면 본 SKILL에 적힌 원칙으로 대체).

- `10_AI_Context/AI_Master_Context.md`
- `02_YouTube/Baseball_Content_Style.md` (또는 `한화 야구 콘텐츠 스타일.md`)
- `02_YouTube/Mali_Persona.md` — **말리 페르소나 정의 문서. 있으면 아래 기본 페르소나보다 우선한다**
- `99_Rules/Adsense_Rules.md` (있으면)

핵심 스타일 원칙(요약):
- 단순 정의 나열 ❌ → 이야기 + 사례 ✅
- 숫자·규칙은 쉽게 설명 (예: "인필드 플라이 = 심판이 미리 '아웃'이라고 선언해주는 보험")
- 선수·심판 비난 ❌ → 상황 분석 ✅
- 과장형 표현 ❌ ("충격", "경악" 남발 금지 — 한 편에 1회 미만)
- 초보 팬도 이해 가능하게, 그러나 고인물도 "오 이건 몰랐네" 할 디테일 1개 이상

## STEP 3 — 말리 페르소나 롱폼 대본 생성

3~5분 영상 기준. 한국어 발화 속도 기준 **분당 약 300자 → 총 1,000~1,500자(발화문 기준)**.
"짧게" 힌트면 3분(약 900~1,000자), "길게" 힌트면 5분(약 1,400~1,500자)에 맞춘다.

### 말리 기본 페르소나 (vault에 `Mali_Persona.md` 없을 때 사용)

- 야구를 20년 본 동네 형/누나 같은 해설 캐릭터
- 친근한 존댓말, 청자는 "여러분"
- 어려운 규칙을 일상 비유로 바꿔 설명하는 게 특기
- 가벼운 유머 1~2회, 그러나 정보 정확성이 항상 우선
- 말버릇 예시: "자, 여기서 재밌는 게 뭐냐면요", "이거 은근 헷갈리시죠?"

### 구조 (단일 파일의 `## 🎙 롱폼 대본` 섹션)

```
## 🎙 롱폼 대본

### 영상 제목 후보
- 궁금증 유발 구조, 예: "보크, 심판도 헷갈리는 이유"
       "인필드 플라이가 없으면 벌어지는 일"

### 0~15초 훅
- 시청자가 나가지 않게 만드는 질문 또는 장면 제시
- 예: "투수가 공을 안 던졌는데 주자가 진루합니다. 왜일까요?"
- KBO 실제 사례로 시작하면 효과적

### 1부 — 이게 뭔가요 (정의)
- 용어를 일상 비유로 먼저, 정식 정의는 그 다음
- 규칙 조항 번호는 화면 자막용으로 괄호 표기

### 2부 — 왜 생겼나요 (기원·역사)
- 스토리텔링으로 풀기 (연도·배경·계기)
- "그 시절엔 이런 일이 있었거든요" 톤

### 3부 — KBO에서 실제로 (사례)
- 리서치의 KBO 사례 1~2개를 장면 묘사하듯
- 판정 논란 사례는 심판 비난 없이 "규칙이 이래서 이렇게 된 것" 구조로

### 4부 — 이건 몰랐죠 (오해 바로잡기)
- 흔한 오해 1~2개 교정
- 고인물용 디테일 1개 배치

### 마무리
- 한 줄 요약 + 다음 편 예고 (관련 용어에서 선택)
- 콜투액션 1줄 ("다음 용어가 궁금하면 구독" 권장)

### 영상 노트 (제작자용)
- 총 길이: 3~5분 목표, 섹션별 예상 시간 표기
- 한글 자막은 후편집으로 (AI 한글 깨짐 주의)
- 규칙 설명 구간은 다이어그램/그래픽 삽입 지점 표시 (예: [그래픽: 주자 위치])
```

### 문체 원칙 (AI 티 나지 않게)

- 구어체 발화문: 실제로 소리 내 읽었을 때 자연스러워야 한다
- 절대 금지: "살펴보겠습니다", "알아볼게요", "정리해보자" 같은 기계 표현
- 문장은 짧게, 한 문장에 정보 하나
- 시청자에게 질문 던지기 2~3회 (리텐션 장치)

## STEP 4 — SEO 패키지 생성

### 구조 (단일 파일의 `## 🏷 SEO` 섹션)

```
## 🏷 SEO

### 제목 (3개 후보 → 추천 1개 표시)
- 40자 이내, 검색 키워드({용어}) 앞쪽 배치 + 궁금증 유발
- ✅ 추천: 클릭률 가장 높을 것으로 보이는 1개
- 예시:
  - 보크가 뭐길래 심판도 헷갈릴까 | 야구사전
  - 보크 완벽 정리, 3분이면 끝납니다

### 설명 (유튜브 설명란)
- 첫 두 줄에 핵심 키워드 + 영상 요약 (접힘 전 노출 영역)
- 이어서 타임스탬프 틀 (00:00 훅 / 00:15 정의 / ... 형태)
- 마지막에 해시태그 3~5개

### 태그 (10~15개)
- 용어 자체, 용어 변형 표기, "야구 규칙", "야구 용어", "KBO",
  사례에 등장한 팀·선수명, 채널명 관련 태그

### 썸네일 텍스트
#### 메인 문구 (3개 후보 → 추천 1개 표시)
- 짧고 강한 8~12자, 예: "심판도 헷갈린다"
#### 서브 문구 (3개 후보 → 추천 1개 표시)
- 메인 보완 궁금증 유도 12~18자, 예: "보크의 진짜 기준"
#### 구성 메모
- 배경·색감·배치 제안 (예: 심판 콜 장면 + 강한 대비)
- AI 한글 텍스트는 반드시 후편집
```

### 저장 직후 필수 검증

```powershell
Get-ChildItem "C:\Users\user\Documents\Joy_SecondBrain\50_Outputs\야구사전\" -Filter "*{용어}*" | Select-Object Name, Length
```

확인 항목:
- [ ] `{YYYY-MM-DD}_{용어}.md` 파일이 존재하는가?
- [ ] 파일 크기가 0이 아닌가?
- [ ] 세 섹션(🔍/🎙/🏷)이 모두 본문에 들어 있는가?
- [ ] 파일명의 날짜가 오늘 날짜와 일치하는가?

파일이 누락됐으면 즉시 재생성 후 보고한다.
Obsidian은 파일 생성 후 캐시 새로고침(Ctrl+Shift+R)이 필요할 수 있으니 안내한다.

## 파일명 규칙 상세

- 경로: `50_Outputs/야구사전/{YYYY-MM-DD}_{용어}.md` (폴더 없으면 생성)
- 날짜: 항상 `YYYY-MM-DD` 형태 (정렬 편의)
- 용어: 한글 그대로, 띄어쓰기는 언더스코어로 (예: `인필드_플라이`)
- 같은 파일이 이미 있으면 덮어쓰지 말고 `_v2` suffix 부여 후 사용자에게 알린다

## 주의사항

- **사실 검증**: 규칙 조항·연도·사례는 반드시 웹에서 확인. 모를 때 만들지 말고 "확인 필요"로 비워둔다.
- **심판·선수 비난 금지**: 판정 논란 사례도 규칙 설명 관점으로만 다룬다.
- **저작권**: 출처 URL은 리서치 노트에만 기록, 대본에 타 채널 문구를 옮기지 않는다.
- **사용자 vault가 없으면**: 임시 outputs 폴더에 저장하고 사용자에게 옮길 위치 안내.
- **사용자가 한 가지만 원하면**: "대본만", "리서치만", "SEO만" 단서가 있으면 해당 결과물만 만든다.
  단, "대본만"이어도 STEP 1 리서치는 내부적으로 수행한다 (사실 검증 없이는 대본을 쓰지 않는다).
